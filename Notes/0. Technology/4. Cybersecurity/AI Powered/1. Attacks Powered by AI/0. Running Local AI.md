# Why run AI locally?

Correr modelos de AI localmente con herramientas como [LM Studio](https://lmstudio.ai/) y [Anything LLM](https://anythingllm.com/) no es solo una cuestión de *privacidad*, en un contexto ofensivo cambia bastante el modelo o perativo. Esto gracias a que:

1. **Aislamiento total de datos**: Cuando trabajamos con información sensible de clientes, como podría nser dumps de credenciales, outputs de herramientas como nmap, bloodhound, burp, etc., código de infraestructura sensible, infraestructura interna, etc. Mandar esta información a APIs externas rompe completamente el modelo de confianza y privacidad. En cambio, utilizar de manera local la AI nos permite:
	- Evitar exfiltración involuntaria hacia terceros.
	- Eliminar riesgos legales/contractuales.
	- Operar en redes air-gapped.

2. **Customización agresiva del modelo**: Al utilizar la AI de manera local, podemos hacer cosas que en cloud resultarian limitadas tal como:
	- *Fine-tuning* (*LoRA*): Entrenar o adoptar modelos con exploits históricos, playbooks propios, TTPs tipo MITRE ATT&CK, payloads específicos, obteniendo un resultaod tal que el modelo piensa de la manera más esperada y personalizada y no como uno generalista.
	- *RAG* (*Retrieval Augmented Generation*): La posibilidad de indexar reportes previos, metodologías internas, cheatsheets, CVEs explotables en el entorno con el fin de poder realizar preguntas tales como "Tengo este servicio expuesto + este banner.... qué vectores priorizo?" o la posibilidad de pedir descripiciones para reportes adaptados a nuestra metodología.

3. **Eliminación de restricciones**: Los modelos SaaS tienen filtros, rate limits, planes, logging, detección de abuso, políticas de seguridad y temas sensibles para hablar. Al utilizar la AI de manera local poseemos cero restricciones, tenemos la posibilidad de explorar cadenas de ataques completas sin tópicos sensibles, así como generar payloads sin censura entre otrccas cosas.

4. **Integración directa con tools ofensivas**: Desde el punto de vista téccnico, una de las mejores mejoras de cara a utilizar la AI de manera local, es la posibilidad de conectar el LLM con:
	- Scrips propios.
	- Herramientas o suites de software ofensivo típicas.
	- APIs internas.
	- Pipelines automatizados.

5. **Latencio y Throughput controlado**: En este caso una de las mejoras significativas es la falta de necesidad de utilizar Internet, no contar con rate limits y teniendo respuestas consistentes.

6. **Persistencia contextual real**: En si, en la AI de tipo SaaS el contexto que almacena la AI puede resultar limitado en planes gratis, en local podemos mantener estados largas (Una campaña completa).

En resumen, instalar AI de manera local trae en comparación a utilizar la que acostumbramos como SaaS muchisimas ventajas, sin contar la capaidad de operar en entornos restringidos, reducir la superfiie de detección ya que no hay tráfico a APIs externas, pdemos experimentar con modelos especializados tal que utilizar modelos cchicos ultra rápidos para parsing, modelos grandes para razonamiento, modelos entrenados en código (Tipo Code LLMs) y combinarlos como pipeline.

```python
from fastapi import FastAPI, Request
import httpx, uuid, time

app = FastAPI()

# MOdel 1, in chargo to parse normal human input into one more technincal
PARSER_URL = "http://localhost:1236/v1/chat/completions"
# Model 2, in charge to reason
REASONER_URL = "http://localhost:1236/v1/chat/completions"

PARSER_PROMPT ="""
You are a prompt pre-processor. Your only job is to rewrite the user's raw input into a well-structured technical query optimized for a reasoning model.

Rules:
- Preserve the user's original intent exactly. Never add assumptions.
- Identify the core technical question and make it explicit.
- Add relevant context if it's clearly implied (e.g. "how does X work" → specify domain if obvious).
- Format the output as a clean, direct technical question or instruction.
- If the user's input is already well-formed, return it with minimal changes.
- Output only the rewritten prompt. No explanations, no preamble.
"""

async def call_llm(url, messages, model, temperature=0.2):

    async with httpx.AsyncClient(timeout=300) as client:

        r = await client.post(url, json={
            "model": model,
            "messages": messages,
            "temperature": temperature,
            "stream": False
        })
        r.raise_for_status()
        return r.json()
        
    
@app.post("/v1/chat/completions")
async def chat_completions(req: Request):
    data = await req.json()
    messages = data.get("messages", [])

    ORIGINAL_RAG_SYSTEM_PROMPT = next(
        (m["content"] for m in messages if m["role"] == "system"),
        ""
    )

    # DEBUG
    print(f"\n\n\t[+] DEBUG (OG REQ): \n{data}\n\n")

    if not messages:
        return {"error": "No messages provided"}

    # ==========================
    #       PARSER STAGE
    # ==========================
    user_input = next(
        (m["content"] for m in reversed(messages) if m["role"] == "user"),
        ""
    )

    parser_messages = [
        {
            "role": "system",
            "content": PARSER_PROMPT
        },
        {
            "role": "user",
            "content": user_input
        }
    ]    

    # DEBUG
    print(f"\n\n\t[!] ===== DEBUG (PARSER REQ BODY) ====\n {parser_messages}\n\n")

    parser_response = await call_llm(PARSER_URL, parser_messages, "llama-3.2-3b-instruct")
    parsed = parser_response["choices"][0]["message"]["content"]

    # DEBUG
    print(f"\n\n\t[!] ===== DEBUG (PARSER RES BODY) ====\n {parsed}\n\n")


    # ===========================
    #       REASON STAGE
    # ===========================

    reasoning_messages = [
        {
            "role": "system",
            "content": ORIGINAL_RAG_SYSTEM_PROMPT
        }
    ] + messages + [
        {
            "role": "user",
            "content": f"\n\n[PARSED]\n{parsed}"
        }
    ]

    # DEBUG
    print(f"\n\n\t[!] ===== DEBUG (REASONER REQ BODY) ====\n {reasoning_messages}\n\n")

    reasoning_response = await call_llm(REASONER_URL, reasoning_messages, "deepseek/deepseek-r1-0528-qwen3-8b")
    final_answer = reasoning_response["choices"][0]["message"]["content"]

    # DEBUG
    print(f"\n\n\t[!] ===== DEBUG (REASONER RES BODY) ====\n {final_answer}\n\n")


    # =========================
    #    OPEN AI OUTPUT GEN
    # =========================
    return {
        "id": f"chatcmpl-{uuid.uuid4()}",
        "object": "chat.completion",
        "created": int(time.time()),
        "model": "Dobliuw-Gateway",
        "choices": [
            {
                "index": 0,
                "message": {
                    "role": "assistant",
                    "content": final_answer
                },
                "finish_reason": "stop"
            }
        ],
        "usage": {
            "prompt_tokens": 0,
            "completion_tokens": 0,
            "total_tokens": 0
        }
    }
    
@app.get("/v1/models")
async def show_models(req: Request):
	
	return {
        "data": [
            {
                  "id": "dobliuw/dobliuw_custom_pipeline",
                  "object": "model",
                  "owned_by": "organization_owner"
             }
        ],
        "object": "list"
    }
```

```txt
You are a professional cybersecurity vulnerability report writer assistant.

Your sole purpose is to generate high-quality security vulnerability reports based ONLY on the provided context (RAG input, retrieved documents, and user-provided notes or templates).

You must NOT use external knowledge unless explicitly present in the provided context. If information is missing, you must explicitly state it as "Not observed in provided data" or "Insufficient evidence in provided sources".

---

## PRIMARY OBJECTIVE

Transform raw security findings, technical notes, or scan outputs into structured, professional vulnerability reports suitable for clients, bug bounty platforms, or internal red team documentation.

Your output must follow the closest possible structure and style of the provided report templates in the context. If multiple templates exist, infer the most consistent pattern and prioritize it.

---

## STRICT BEHAVIOR RULES

1. STRICTLY USE PROVIDED CONTEXT ONLY
   - Do not hallucinate vulnerabilities, CVSS scores, exploit steps, or impact details not present in the context.
   - Do not assume system configurations or environments unless explicitly stated.

2. TEMPLATE ADHERENCE IS CRITICAL
   - If report examples or templates are provided in context, replicate:
     - structure
     - tone
     - section order
     - formatting style
   - You may adapt wording slightly for clarity, but NOT structure.

3. PROFESSIONAL TONE
   - Use formal, technical, and precise language.
   - Avoid conversational explanations.
   - Write as a senior security consultant delivering a final report.

4. NO GENERIC SECURITY ADVICE
   - Only include mitigation steps that are relevant to the described vulnerability.
   - Do NOT add unrelated best practices unless they directly map to the issue.

---

## REPORT STRUCTURE (DEFAULT IF NO TEMPLATE PROVIDED)

If no explicit template is found in context, use the following structure:

### 1. Title
Clear and concise vulnerability name.

### 2. Severity
If not provided, infer cautiously from context. If uncertain, state "Undetermined based on provided data".

### 3. Affected Component
Specify system, endpoint, service, or module.

### 4. Description
Technical explanation of the vulnerability based strictly on evidence.

### 5. Impact
Describe realistic consequences based only on available context.

### 6. Evidence
Include references, logs, payloads, or outputs provided in the context.

### 7. Steps to Reproduce
Only if reproduction steps exist in the context. Otherwise state:
"Reproduction steps not fully provided in the source material."

### 8. Recommendation / Mitigation
Provide precise and technically accurate remediation steps tied to the issue.

### 9. References (if available)
Include only references present in the context.

---

## QUALITY REQUIREMENTS

- Ensure consistency across all generated reports.
- Avoid redundancy between sections.
- Prioritize clarity over verbosity.
- Maintain enterprise-grade vulnerability reporting standards.
- Ensure each report could be directly delivered to a client or included in a pentest PDF.

---

## HANDLING MULTIPLE FINDINGS

If multiple vulnerabilities are present:
- Separate each finding clearly.
- Do NOT merge unrelated issues.
- Maintain consistent structure per finding.

---

## FAILURE CONDITIONS

If the input data is insufficient:
- Do NOT fabricate missing information.
- Clearly state limitations of available data.
- Request missing technical details only if absolutely required.

---

You are not a general assistant.
You are a vulnerability report generation engine operating on structured security intelligence.
```
