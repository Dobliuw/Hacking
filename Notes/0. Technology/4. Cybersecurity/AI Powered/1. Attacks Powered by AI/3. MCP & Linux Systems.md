# Model Context Protocol (MCP)

El **Model Context Protocol** (**MCP**) es un estándar abierto creado por *Anthropic* que permite a los modelos de IA conectarse de forma segura y estandarizada con herramientas, datos y servicios externos. La idea central es simple aunque muy potente:

> En lugar de que cada app invente su propia forma de darle contexto a un modelo, MCP define una interfaz común.

MCP se basa en *tres componentes*:

1. **Client MCP** (*Host*): La aplicación de IA que usa el modelo, sea app o agente. Ejemplo Claude Desktop, Cursor, Warp, etc.

2. **Servidor MCP**: Expone capacidades externas en formas de proceso independiente que envuelve un servicio externo como GitHub, base de datos, filesystem, etc. y lo hace accesible vía MCP de forma estructurada:
	- Tools (Acciones - [[4. Model Context Protocol (MCP)#Tools]]).
	- Resources (Datos - [[4. Model Context Protocol (MCP)#Resources]]).
	- Prompts (Plantillas - [[4. Model Context Protocol (MCP)#Prompts]]).

3. **Modelo** (*LLM*): Modelo de inteligencia artificial, generalmente consumido por el *Client MCP* encargado de recibir el contexto ensamblado por el *Client MCP*, incluyendo el mensaje del usuario, junto con los resultados de *Tools* y *Resources* del servidor y genera la respuesta.

Con MCP, el modelo puede conectarse a servicios como:

- Google Drive / OneDrive.
- Gmail / Outlook.
- Google Calendar.
- Github.
- Bases de datos.

Es *open source* y esta siendo adoptado por la industria, lo que lo convierte en un estándar emergente para hacer que los modelos de IA sean verdaderamente útiles en flujos de trabajo reales, con acceso a información actualizada y herramientas del mundo real. En resumen, MCP es la "infraestructura" que le da a los modelos los recursos y habilidades necesarios para operar como agentes capaces de interactuar con el mundo real.

En resumen: 

```txt
User prompt 
  ↓
Client MCP ----> La App (Ej. OpenCode) mete en el contexto del LLM el Prompt + Tools y Resources disponibles.
  ↓
 LLM ----------> Lee el prompt + capabilities
  ↓
 LLM ----------> Decide si necesita llamar alguna Tool.
  ↓
Client MCP ----> Ejecuta la llamada al Server MCP
  ↓
Client MCP ----> Devuelve resultado a LLM
  ↓
LLM -----------> Decide si necesita más llamadas o ya puede responder.
  ↓
Respuesta final
```

Así mismo, los comandos listados en el *Client MCP* tipo `/make-something`son **Prompts** ([[4. Model Context Protocol (MCP)#Prompts]]) predefinidos para realizar una tarea, sin necesidad de que el usuario tenga que ingresar el input que ya guarda el prompt del *Servidor MCP*. Y esto es así, porq ni bien se conecta al Servidor MCP el Client MCP, hace una llamada de descubrimiento para entender y guardar en memoria las *Tools*, *Prompts* y *Resources* disponibles, lo cual se conoce como ==capability negotiation== y ocurre vía *JSON-RPC*.

# Automation Strategy

En este punto, el papel estrategico de la automatización puede verse en cuatro aspectos:

1. **Eficiencia operativa**: Reducción drástica del tiempo en tareas repetitivas: Escaneos, recoleción de datos y análisis preliminares se ejecutan sin intervención manual.
2. **Aceleración de procesos**: Los workflows automatizados procesan miles de endpoints simultáneamente, completando en minutos lo que manualmente requeriría horas.
3. **Reducción de errores**: La ejecución programática elimina fallos humanos en configuraciones, parámetros y secuencias complejas de comandos.
4. **Integración CI/CD**: Incorporacción de pruebas de seguridad en pipelines de desarrollo continuo para validación automática de cada despliegue.

Así mismo en sistemas operativos como *Kali Linux* o *Black Arch*, el enfoque de automatización orientado a ciberseguridad daría lugar a:

1. **Bash scripting** para cadenas de creación de scripts que encadenan múlltiples herramienteas en fujos de trabajo controlados y seuenccias.
2. **Automatización de herramientas core**, ejemplo ejecución programada de herramientas como Nmap, Nikto, SQLmap y Metasploit ccon parámetros optimizados y salidas estructuradas.
3. **Generación de reportes**: Uso de EyeWitness, recon-ng y frameworks personalizados para documentación automática de hallazgos.
4. **Orquestación con Python**: Scripts avanzados que coordinan herramientas, procesan resultados y toman deccisiones basada en datos.
5. **Integración con APIs**: Enriquecimiento de resultados mediante consultas a servicios externos: Threat intelligence, OSINT y bases de vulnerabilidades.

En cambio, la automatización orientada a ciberseguridad en entornos *Windows* daría lugar a cosas como:

1. **Capacidad de batch scripting**:
	- Ejeccución de comandos nativos para diagnóostico: netsh, schtasks, wmic, reg query.
	- Recolección sistemática de logs de eventos y configuraciones del Sistema.
	- Auditorías automatizadas de usuarios, permisos y políticas de grupo.
	- Tareas programadas para monitoreo continuo y respuesta rápida.
2. **Integración híbrida**: Combinación de scripts batch con PowerShell para flujos más sofisticados, aprovechando .NET Framework y acceso a WMI.
