# Components

Un sistema de **Machine Learning** no es solo el modelo, como suele creerse, si no más bien un *pipeline completo de ingeniería de datos*, *entrenamiento*, *evaluación* y *operación*. En la práctica esto se conoce como **ML Lifecycle** o **ML Pipeline**, y es lo que da origen al campo de *MLOps*.

#### Data

Los **datos** representan la *experiencia a partir de la cual el sistema aprende*. En machine learning no se programa explícitamente el comportamiento del sistema; en cambio, el modelo intenta *inferir patrones estadísticos* a partir de ejemplos observados.

Los datasets suelen contener muchos pares de datos que sirven como ejemplos de una relación. Esto implica que los datos son una *representación parcial del mundo real*. Nunca acpturan toda la complejidad del fenómeno, sino una muestra limitada y condicionada por cómo se recolectaron.

Por esto, entender de donde vienen los datos es tan importante como el modelo. Existen varias fuentes típicas:

- **Datos generados por sistemas**: Logs de servidores, registros de sensores, telemetría de aplicaciones, transacciones financieras, etc.
- **Datos recolectados manualmente**: Encuenstas, etiquetado humano, anotaciones para visión o lenguaje.
- **Datos derivados**: Resultados de otros sistemas o modelos previos.
- **Datos públicos o abiertos**: Datasets académicos o corporativos.

En muchos proyectos, esta fase, con el fin de obtener *datos confiables* suele ser la más costosa. Esto implica diseñar pipelines de recolección, sistemas de almacenamiento y mecanismos de validación de dichos datos. 

Conceptualmente, los datos suelen dividirse en dos tipos:

- **Features**: Variables que describen la situación o el objeto que queremos analizar.
- **Labels**: La respuesta o resultado que el modelo debe aprender a predecir.

Un aspecto crítico es que los datos deben representar la *distribución real del problema*. Si el dataset no refleja adecuadamente el mundo donde el modelo se usará, el aprendizaje será incorrecto. Este fenómeno se conoce como **dataset bias** o **distribution mismatch**. Además del problema estructural: Los datos crudos casi nunca están listos para ser usados directamente. Pueden contener errores, incosistencias o estructuras incompatibles con los algoritmos, debido a esto existe la siguiente fase.

#### Pre-Procesing

El **preprocesamiento** es el proceso mediante el cual los *datos curos se transforman en representaciones adecuadas para el aprendizaje automático*. Esta fase es necesaria porque los algoritmos de ML operan sobre estructuras numéricas bien definidas, mientras que los datos reales suelen ser heterogéneso y desordenados.

El preprocesamiento cumple varios objetivos:

1. *Garantizar la calidad de los datos*: Esto implica identificar valores faltantes, corregir incosistencias y eliminar registros corruptos. En datasets grandes es común encontrar duplicados, errores de medición o valores imposibles.

2. *Representar la información de manera útil para el modelo*: Esto se conoce como **feature engineering** y consisten en transformar variables originales en nuevas variables que captures mejor las relaciones relevantes. ejemplo, un timestamp bruto puede transformarse en variables como hora del día, día de la semanao estacionalidad. Estas transformaciones permiten que el modelo capture patrones que de otro modo quedarían ocultos.

3. *Convertir datos no númericos en representacioens matemáticas*: Los algoritmos de aprendizaje trabajan con vectores numéricos, por lo que variables categóricas o textuales deben transformarse mediante técnicas como codificación categórica o embddings.

4. *Normalizar o escalar variables*: Muchas veces una variable tiene valores entre 0 y 1 y otra entre 0 y 10.000, algunos algoritmos pueden volverse inestables o dar más peso a una de ella simplemente por su escala.

5. *Separación de dataset en subconjuntos independientes*: Esto es crucial porque el modelo debe evaluarse con datos que no haya visto durante el entrenamiento. Si se entrenara y evaluara con los mismos datos, el sistema simplemente memorizaría los ejemplos sin aprender aptrones generalizables.

Por todo esto, los dataset suelen dividirse en tres partes conceptuales:

- **Datos de entrenamiento**: Utilizados para ajustar el modelo.
- **Datosde validación**: Utilizados apra ajustar hiperparámetros y seleccionar modelos.
- **Datos de prueba**: Utilizados para estimar el rendimiento final

#### Training

El **entrenamiento** es el proceso mediante el cual el modelo *aprende los parámetros que mejor explican la relación entre los datos de entrada y salidas esperadas*. En las redes neuronales pueden haber millones o incluso billones de parámetros.

Durante el entrenamiento, el sistema busca encontrar el conjunto de parámetroes que minimiza una función de error. Esta función mide la diferencia entre la predicción del modelo y el valor real observado en el dataset.

El entrenamiento consiste entonces en un problema de optimización matemática. Los algoritmos ajustan iterativamente los parámetros del modelo para reducir el error promedio en el dataset.

En muchos modelos modernos se utiliza una técnica llamada *descenso de gradiente*. Esta técnica calcula la dirección en la que la función de error disminye más rápidamente y ajusta los parámetros de esa dirección. El proceso se repite miles o millones de veces hasta que el modelo converge a una solución aceptable.

Además del algortimo de optimización, el entrenamiento también involucra decisiones sobre la arquitectura del modelo y los llamados *hiperparámetros*, los cuales controlan el comportamiento de algoritmo de aprendizaje, como la velocida de aprendizaje, la complejidad del modelo o el tamaño de los batches de datos utilziados en cada iteración.

> [!warning] Cost & Warning
> El entrenamiento puede ser computacionalmente muy costoso, especialmente en modelos modernos de deep learning que requieren grandes cantidades de datos y hardware especializado como GPUs o TPUs.
> 
> Pero incluso si el modelo logra minimizar el error en los datos de entrenamiento, eso no garantiza que funcione bien en situaciones nuevas.... para comprobarlo existe la siguiente fase.

> [!warning] Metrics of train
> Las **metricas** de evaluación de modelos de AI son medidas matemáticas que se utilizan para evluar, comparar y mejorar objetivamente los modelos de inteligencia artifical y aprendizaje automático. Estas métricas pemiten evaluar objetiva y cuantitativamente la eficacia de un modelo para una tarea determinada.
> 
> - *Accuracy*: El accuracy mide proporción total de aciertos pero ignora completamente la distribución de las clases y el tipo de error que el modelo comete. Es por esto que la presición de un modelo puede estar equivocada.
>   
>   Por ej: Un modelo entrenado para detectar malware podría enfrentarse a 10.000 archivos donde únicamente 10 son malware y 9.990 son benignos y determinar que "todos son benignos". El modelo tendría un *99.9%* de acierto, pero sería completamente inútil.
>   
>- *Recall*: Es una métrica de rendimiento fundamental en el aprendizaje automático que mide la capacidad de un modelo para identificar todas las instanacias relevantes dentro de un conjunto de datos. El modelo *determina cuantos de los positivos reales detectó correctamente*.
 
#### Validation

La **validación** evalúa *qué tan bien el modelo generaliza a datos que no formaron parte del entrenamiento*. El objetivo es estimar cómo se comportará el modelo cuando enfrente datos reales en producción.

El problema que se intenta evitar aquí es el **overfitting** o sobreajuste. Esto implica que un modelo con demasiada capacidad puede *aprender patrones muy específicos del dataset de entrenamiento, incluyendo ruido o coincidencias accidentales*. En ese caso, el modelo funciona muy bien con los datos que ya vio, per falla cuando se enfretan a ejemplos nuevos.

La validación permite detectar este problema comparando el rendimiento del modelo en diferentes subconjuntos de datos.

En esta fase también se selecciona el modelo final entre varias alternativas. Es común entrenar múltiples configuraciones del modelo con distintos hiperparámetros y comparar su rendimiento. El modelo elegido es aquel que logra el mejor equilibrio entre precisión y capacidad de generalización.

En sintesis, la validación sirve para:

- Medir rendimiento del modelo.
- Analizar errores.
- Identificar sesgos en el dataset.
- Detectar situaciones en las que ele modelo se podría comportar de manera inesperada.

#### Deployment

El **deployment** marca la transición entre un experimiento de laboratorio y un sistema real. En esta etapa *el modelo entrenado se integra dentro de una aplicación o infraestructura que le permite ser consumido*.

Esto implica resolver varios problemas de ingeniería. El modelo debe cargarse en memoria, recibir datos de entrada en el formato adecuado, producir predicciones y devolverlas a otros sistemas que tomarán decisiones basadas en ellas.

En muchos casos el modelo se expone como un servicios accesible mediante API. Otras veces se integra diectamente dentro de una aplicación como ocurren con modelos que funcionan en dispositivos móviles o sistemas embebidos

> [!tip] API?
> Por eso recomiendo muchisimo tras ataques y producciones que tuve de ver, probar *prompt injecting* u otro tipo de ataques a AI en parametros o endpoints de APIs, ya que muchas veces suelen ser conectados mediante este flujo de datos de entrada sin que sea perceptible.

El deployment también implica considerar factores como latencia, escalabilidadad y confiabilidad. Un modelo puede ser muy preciso pero inutilizable si tarda demasiado en producir una predicción o si requiere más recursos de los disponibles.

Además, esta fase debe asegurar que el flujo de datos en producción sea *consistente con el utilizado durante el entrenamiento*. Si los datos llegan en un formato distinto o pasan por transformaciones diferentes, el modelo puede producir resultados erróneos.

#### Monitoring

Una vez desplegado (*Deployment*), el modelo deja de ser un artefacto estático y pasa a formar parte de un sistema dinámico que interactúa con el mundo real. La fase de **monitoreo** existe porque *las condiciones del mundo cambian constantemente*.

Los modelos de machine learning se entrenan con datos históricos. Pero con el tiempo pueden aparecer nuevos comportamientos, cambios en los usuarios o nuevas condiciones que alteren la distribución de los datos.

Cuando esto ocurre, el modelo puede empezar a perder precisión. Este fenómeno se conoce como **data drift** o **concept drift**.

El monitoreo consiste en observar continuamente el comportamiento del modelo en producción para detectar estos cambios. Inlcuyendo analizar la distribución de los datos de entrada, medir el rendimiento de las predicciones y detectar anomalías en el sistema.

Si el rendimiento cae por debajo de ciertos umbrales, se inicia un proceso de **reentrenamiento**, donde el modelo se actualiza utilizando datos más recientes.

> [!info] Life Cycle
> En conjunto, todas estas fases forman un **ciclo continuo**. un sistema de machine learning no se construye una vez y se deja funcionando indefinidamente. Más bien evoluciona con el tiempo, adaptándose a nuevos datos y a cambiuos en el entorno.
